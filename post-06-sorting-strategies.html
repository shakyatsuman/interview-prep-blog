Sorting Strategies

Sorting is not just a prerequisite — it is the solution. Recognizing when a sort transforms an intractable problem into a trivial one is one of the highest-leverage skills in an interview.

⏱ 5 min read 🗂 Fundamentals 🔑 Java 8 📈 Intermediate → Advanced

Most sorting coverage in interview prep treats sorting as a preprocessing step — something you do before applying another technique. This post flips that perspective: sorting is often the technique itself. Understanding why, and recognizing the problems where it applies, is what separates engineers who reach for the right tool from those who over-engineer a solution.

The Core Idea

Sorting imposes order on a collection. Once elements are ordered, many problems become trivial: adjacency reveals duplicates, proximity reveals intervals to merge, and relative position enables two-pointer scans. The cost is always O(n log n) time and typically O(1) or O(n) space. The question to ask is: does imposing order make the problem structurally simpler? If yes, sort first.

Fig 1. One sort pass makes duplicates adjacent and intervals mergeable — unlocking O(n) solutions to problems that would otherwise require O(n²).

Two Sort-First Patterns

Pattern 1 — Sort Then Scan

Sort the input, then solve in a single pass — whether that pass is a linear scan, a two-pointer sweep, a greedy selection, or a binary search. Adjacency after sorting is what makes all of these possible. The sort costs O(n log n); the subsequent pass is O(n), so the total is O(n log n). Examples: merging overlapping intervals, removing duplicates, finding pairs that satisfy a sum constraint.

Pattern 2 — Custom Comparator

The default lexicographic or numeric sort order is wrong for the problem. Define a custom comparator that encodes the problem's specific ordering rule. The comparator is the insight — once you define it correctly, sorting does all the work. Examples: arranging numbers to form the largest value, sorting tasks by a custom priority.

Example 1 — Merge Intervals

LeetCode #56 · Medium · Sort then scan

Given an array of intervals where intervals[i] = [start, end], merge all overlapping intervals and return the result.

Example

[[1,3],[2,6],[8,10],[15,18]] → [[1,6],[8,10],[15,18]]
[1,3] and [2,6] overlap (3 ≥ 2) so they merge to [1,6].

Without sorting, detecting which intervals overlap requires O(n²) comparisons. After sorting by start time, overlapping intervals are guaranteed to be adjacent — a single linear scan suffices. Two intervals overlap when curr.start <= prev.end; merge by extending prev.end = max(prev.end, curr.end).

Fig 2. After sorting by start time, overlapping intervals are always adjacent. A single left-to-right scan merges them in O(n).

Java 8public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);  // sort by start time

    List<int[]> result = new ArrayList<>();
    result.add(intervals[0]);

    for (int i = 1; i < intervals.length; i++) {
        int[] curr = intervals[i];
        int[] last = result.get(result.size() - 1);

        if (curr[0] <= last[1]) {
            last[1] = Math.max(last[1], curr[1]);  // merge: extend end
        } else {
            result.add(curr);                       // no overlap: append
        }
    }

    return result.toArray(new int[0][]);
}
// Time: O(n log n)  |  Space: O(n)
Key Insight

The comparator (a, b) -> a[0] - b[0] sorts by start time. After this sort, any two intervals that overlap must be adjacent in the array. A single pass then merges them greedily left to right.

Example 2 — Largest Number

LeetCode #179 · Medium · Custom comparator

Given a list of non-negative integers, arrange them so that they form the largest number and return it as a string.

Example

nums = [3, 30, 34, 5, 9] → "9534330"
nums = [10, 2] → "210"

The trick: numeric or lexicographic sort order is wrong here. The correct ordering asks: given two numbers A and B, does AB (A concatenated with B) form a larger number than BA? Define the comparator to answer exactly this question, and Java's sort does the rest.

Fig 3. The comparator (b + a).compareTo(a + b) encodes the problem's ordering rule. Once defined, sorting handles the rest in O(n log n).

Java 8public String largestNumber(int[] nums) {
    String[] strs = new String[nums.length];
    for (int i = 0; i < nums.length; i++) {
        strs[i] = String.valueOf(nums[i]);
    }

    // comparator: prefer the order that forms the larger concatenation
    Arrays.sort(strs, (a, b) -> (b + a).compareTo(a + b));

    if (strs[0].equals("0")) return "0";  // edge case: all zeros

    StringBuilder sb = new StringBuilder();
    for (String s : strs) sb.append(s);
    return sb.toString();
}
// Time: O(n log n · k)  k = avg digit length  |  Space: O(n)
Why this comparator works

If (b + a) > (a + b) as strings, b should come before a. Java's compareTo on strings compares lexicographically, which works correctly here because both concatenations always have the same total number of digits and thus the same length.

When to Reach for Sort First

  • The input is unsorted and you find yourself needing to compare every pair — sorting eliminates the need
  • "Merge overlapping intervals" or any problem where adjacency after sorting reveals relationships
  • "Arrange numbers to form the largest/smallest value" — almost always a custom comparator
  • A greedy strategy exists but only works on ordered input
  • You want to use two pointers or binary search but the array is not sorted
