Prefix Sum

Precompute once, query in O(1). A single preprocessing pass eliminates repeated range scans and underpins problems across sliding window, DP, and beyond.

⏱ 5 min read 🌾 Fundamentals 🔑 Java 8 📈 Intermediate → Advanced

A range sum query asks: what is the sum of elements from index i to index j? The naive answer scans every element in that range — O(n) per query. If the array is fixed and queries are frequent, that cost compounds rapidly. Prefix sum solves this by spending O(n) once during preprocessing, then answering every subsequent query in O(1).

Prerequisites

Post #1 (Arrays & Hashing) — familiarity with arrays and O(1) index access. No further prerequisites.

The Core Idea

Given an array arr of length n, construct a prefix sum array prefix of length n+1 where prefix[i] stores the sum of all elements from index 0 up to index i−1. With this structure, the sum of any subarray arr[i..j] is simply prefix[j+1] - prefix[i] — a single subtraction.

Fig 1. prefix[0] = 0 (sentinel). Each subsequent cell accumulates the running total of arr. Building this array costs O(n) once.

The leading 0 at prefix[0] is a sentinel that simplifies the range formula. It means every subarray sum — including those starting at index 0 — can be computed uniformly as prefix[j+1] - prefix[i] without any special-casing.

Two Conventions in the Wild

Prefix sum is defined two ways in textbooks and online resources — it is worth knowing both.

Convention A — 1-indexed sentinel (used in this series)
prefix[0] = 0  ·  prefix[i] = prefix[i−1] + arr[i−1]
Range query: sum(arr[i..j]) = prefix[j+1] − prefix[i]
The extra leading 0 means every range query uses the same formula with no edge cases.

Convention B — 0-indexed inclusive
prefix[0] = arr[0]  ·  prefix[i] = prefix[i−1] + arr[i]
Range query: sum(arr[i..j]) = prefix[j] − prefix[i−1]  (but when i=0, return prefix[j])
Common in textbooks and 1-indexed competitive programming problems, but requires a special case at the left boundary.

The two are mathematically equivalent. This series uses Convention A throughout because the uniform formula is less prone to off-by-one errors under interview pressure.

The Range Sum Query

Sum of arr[i..j] = prefix[j+1] − prefix[i]

Fig 2. Query arr[2..5]: subtract prefix[2]=4 from prefix[6]=23. The answer 19 is computed in one operation regardless of range size.

Java 8// Build prefix sum array — O(n) time, O(n) space
public int[] buildPrefix(int[] arr) {
    int[] prefix = new int[arr.length + 1]; // prefix[0] = 0 sentinel
    for (int i = 1; i <= arr.length; i++) {
        prefix[i] = prefix[i - 1] + arr[i - 1];
    }
    return prefix;
}

// Range sum query — O(1) per query
public int rangeSum(int[] prefix, int i, int j) {
    return prefix[j + 1] - prefix[i]; // sum of arr[i..j] inclusive
}

Example 1 — Range Sum Query – Immutable

LeetCode #303 · Easy

Design a data structure that accepts an integer array at construction time and answers multiple sumRange(i, j) queries, each returning the sum of elements between indices i and j inclusive.

Example

arr = [-2, 0, 3, -5, 2, -1]
sumRange(0, 2) → 1   (−2 + 0 + 3)
sumRange(2, 5) → −1   (3 − 5 + 2 − 1)
sumRange(0, 5) → −3   (entire array)

Build the prefix array once in the constructor. Each sumRange call is then a single subtraction — O(1) regardless of range size or number of queries.

Java 8class NumArray {
    private int[] prefix;

    public NumArray(int[] nums) {
        prefix = new int[nums.length + 1];
        for (int i = 1; i <= nums.length; i++) {
            prefix[i] = prefix[i - 1] + nums[i - 1]; // accumulate
        }
    }

    public int sumRange(int left, int right) {
        return prefix[right + 1] - prefix[left]; // O(1) query
    }
}
// Constructor: O(n)  |  sumRange: O(1)  |  Space: O(n)

Example 2 — Find Pivot Index

LeetCode #724 · Easy · LC75 Problem

Given an integer array, return the leftmost pivot index where the sum of all elements to the left equals the sum of all elements to the right. Return −1 if no such index exists.

Example

nums = [1, 7, 3, 6, 5, 6] → 3
Left sum = 1+7+3 = 11  ·  Right sum = 5+6 = 11

With the total sum in hand, the right sum at any index is total − leftSum − nums[i] (everything to the right, excluding the pivot). The left sum is the running prefix. No nested loop needed.

Fig 3. Pivot index trace. leftSum grows by nums[i] on each iteration; rightSum is derived from total in O(1).

Java 8public int pivotIndex(int[] nums) {
    int total = 0;
    for (int num : nums) total += num; // compute total sum

    int leftSum = 0;
    for (int i = 0; i < nums.length; i++) {
        int rightSum = total - leftSum - nums[i];
        if (leftSum == rightSum) return i;  // pivot found
        leftSum += nums[i];                  // advance left sum
    }

    return -1; // no pivot exists
}
// Time: O(n)  |  Space: O(1)  (no prefix array needed here)
Insight

Pivot Index can be solved with O(1) space by computing the running left sum directly — no prefix array needed. This is a common interviewer follow-up: can you reduce space? The answer is yes, because we access prefix values sequentially, never at random.

Universal Template

The two canonical prefix sum patterns:

Java 8 — Pattern A: Prefix Array// Build once, random-access range sums in O(1)
int[] prefix = new int[arr.length + 1]; // prefix[0] = 0
for (int i = 1; i <= arr.length; i++) {
    prefix[i] = prefix[i - 1] + arr[i - 1];
}
// Range sum arr[i..j] inclusive:
int rangeSum = prefix[j + 1] - prefix[i];
Java 8 — Pattern B: Running Sum (O(1) space)// Sequential access only — no random lookup needed
int total = 0;
for (int num : arr) total += num;

int leftSum = 0;
for (int i = 0; i < arr.length; i++) {
    int rightSum = total - leftSum - arr[i];
    // use leftSum and rightSum here
    leftSum += arr[i];
}

When to Reach for Prefix Sum

Recognize these signals in a problem statement:

  • "Sum of elements between index i and j" or any range sum query
  • Multiple queries on a fixed (immutable) array
  • "Left sum equals right sum" or finding a balance/pivot point
  • "Product of array except self" — prefix product is a direct extension
  • Any problem where recomputing a running aggregate on every iteration feels redundant

The key realization — prefix[j] − prefix[i] = k means prefix[i] = prefix[j] − k — makes it a strong test of whether both patterns are truly internalized.
