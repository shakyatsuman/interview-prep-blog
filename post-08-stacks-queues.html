Stacks & Queues

Last in, first out. First in, first out. Two access patterns that unlock a surprising range of problems — especially the monotonic stack, which solves an entire family of next-greater-element problems in O(n).

⏱ 5 min read 🗂 Data Structures 🔑 Java 8 📈 Intermediate → Advanced

A stack is a collection with one rule: the last element added is the first removed (LIFO). A queue enforces the opposite: first in, first out (FIFO). Both provide O(1) push and pop. Their power in interviews comes not from their basic operations but from what they remember — a stack remembers the order elements arrived; a monotonic stack remembers the last element that was larger or smaller than the current one. That selective memory reduces problems that seem to need O(n²) comparisons down to O(n).

Three Core Patterns

Stack — Matching & Validation

Push opening elements onto the stack. When a closing element arrives, pop and verify it matches. If the stack is empty at the end, the input is valid. This handles parenthesis matching, HTML tag validation, and any problem involving nested structure.

Monotonic Stack — Next Greater / Smaller Element

Maintain a stack that is always sorted (either increasing or decreasing). When a new element violates the order, pop elements until the invariant is restored — each popped element has found its answer. One pass, O(n) total. Each element is pushed once and popped at most once — a fixed budget of 2n operations regardless of how the while loop distributes them. Amortized O(1) per element.

Queue / Deque — Sliding Window Maximum

Maintain a deque of indices whose values are in decreasing order. When the window slides, remove the front if it falls outside the window, and remove the back while the back value is smaller than the incoming element. The front always holds the window maximum.

Example 1 — Valid Parentheses

LeetCode #20 · Easy · Stack matching

Given a string containing only '(', ')', '{', '}', '[', ']', determine if the input string is valid. Every opening bracket must be closed by the same type in the correct order.

Example

"()[]{}"` → true
"([)]"  → false
"{[]}"  → true

Fig 1. Valid Parentheses trace on "{[]}". Each opening bracket is pushed; each closing bracket pops and verifies. Empty stack at end = valid.

Java 8public boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();

    for (char c : s.toCharArray()) {
        if (c == '(' || c == '{' || c == '[') {
            stack.push(c);                  // opening: push
        } else {
            if (stack.isEmpty()) return false;
            char top = stack.pop();
            if (c == ')' && top != '(') return false;
            if (c == '}' && top != '{') return false;
            if (c == ']' && top != '[') return false;
        }
    }
    return stack.isEmpty();  // valid only if all brackets matched
}
// Time: O(n)  |  Space: O(n)

Example 2 — Daily Temperatures

LeetCode #739 · Medium · Monotonic stack

Given an array of daily temperatures, return an array where answer[i] is the number of days you have to wait after day i to get a warmer temperature. If no warmer day exists, answer[i] = 0.

Example

temperatures = [73, 74, 75, 71, 69, 72, 76, 73]
answer       =  [ 1, 1, 4, 2, 1, 1, 0, 0]

Maintain a monotonic decreasing stack of indices. When a temperature is warmer than the index at the top of the stack, that index has found its answer: current_index − stack_top_index. Pop it, record the gap, and repeat. Each index is pushed once and popped at most once — O(n) total despite the inner loop.

Fig 2. Monotonic decreasing stack trace. Each pop resolves one answer in O(1). i=6 (temp=76) resolves indices 5, 2 in one sweep — total work is O(n), not O(n²).

Java 8public int[] dailyTemperatures(int[] temperatures) {
    int n = temperatures.length;
    int[] answer = new int[n];
    Deque<Integer> stack = new ArrayDeque<>();  // stores indices

    for (int i = 0; i < n; i++) {
        // pop indices whose temperature is beaten by current
        while (!stack.isEmpty() && temperatures[i] > temperatures[stack.peek()]) {
            int idx = stack.pop();
            answer[idx] = i - idx;  // days until warmer
        }
        stack.push(i);
    }
    return answer;  // unpopped indices remain 0
}
// Time: O(n)  |  Space: O(n)
Key Insight

The stack stores indices, not values. When we pop, we compute current_index − popped_index to get the wait. Each index is pushed once and popped at most once, so the total work across all iterations is O(n) — not O(n²) despite the while loop.

Example 3 — Sliding Window Maximum

LeetCode #239 · Hard · Monotonic deque

Given an integer array and a window size k, return the maximum value in each sliding window as it moves from left to right.

Example

nums = [1, 3, -1, -3, 5, 3, 6, 7], k = 3
→ [3, 3, 5, 5, 6, 7]

Maintain a monotonic decreasing deque of indices. On each step: remove the front if it has slid out of the window, remove the back while the back value is smaller than the incoming element (it can never be a future maximum), then add the current index. Once the first full window is reached, the front of the deque is always the maximum.

Fig 3. Monotonic deque trace (k=3). Highlighted values (3, 5, 6, 7) are the window maxima. The deque front always holds the index of the current maximum.

Java 8public int[] maxSlidingWindow(int[] nums, int k) {
    int n = nums.length;
    int[] result = new int[n - k + 1];
    Deque<Integer> deque = new ArrayDeque<>();  // stores indices

    for (int i = 0; i < n; i++) {
        // remove front if outside window
        if (!deque.isEmpty() && deque.peekFirst() <= i - k) {
            deque.pollFirst();
        }
        // remove back while smaller than current (can never be max)
        while (!deque.isEmpty() && nums[deque.peekLast()] <= nums[i]) {
            deque.pollLast();
        }
        deque.offerLast(i);

        if (i >= k - 1) {
            result[i - k + 1] = nums[deque.peekFirst()];  // front = max
        }
    }
    return result;
}
// Time: O(n)  |  Space: O(k)
Two Conditions, Two Ends

The deque does two different things at two different ends: front — evict indices that have slid out of the window (size check); back — evict indices whose values are smaller than the incoming element (they can never become a maximum while this element is in the window). Keeping these two operations distinct is the key to implementing this cleanly.

When to Reach for Stack / Queue Patterns

  • "Valid" or "balanced" structure with nested brackets or tags — stack matching
  • "Next greater element", "next smaller element", "days until warmer" — monotonic stack
  • "Previous greater element" — same monotonic stack, iterate right to left
  • "Maximum in sliding window of size k" — monotonic deque
  • Iterative tree or graph traversal replacing recursion — explicit stack
